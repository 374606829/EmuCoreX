# EmuCoreX 安卓新增功能清单

本文档供开发团队实现两项能力：**（A）金手指 PC 版全量移植（`Patch.cpp` 逻辑原封对齐，仅做架构薄适配）**、**（B）贴图提取与替换（含对标金手指页的贴图专属 UI）**。实现原则：**核心逻辑优先复用 / 同源迁移现有 PCSX2 / EmuCoreX 原生代码路径**，贴图与金手指均避免平行实现、重复造轮子。

---

## A. 金手指：PC 版 **全量** 移植（不要求改安卓 UI）

### A.0 硬性要求（范围与原则）

- **金手指须全量移植**：以 `D:\PS2\pcsx2\pcsx2` 中 **`Patch.cpp`（及与之配套的 `Patch.h` 对外声明、数据结构）所承载的代码逻辑为唯一真相来源**，在安卓侧实现 **与 PC 行为等价** 的完整能力（含但不限于：各类 `patch=` / `extended` 处理、动态补丁、指针链 / 值链、F 门、随机数与扩展数据类型、平台 pnach 合并与热重载等 **PC 分支已实现的全部逻辑**）。
- **逻辑层原封不动**：`Patch.cpp` 中的算法、状态机、解析规则、边界条件、与 `GameDatabase` / 配置 / 其它模块的交互语义，应 **逐段迁移并保持一致**；**禁止**在安卓上重写一套「简化版」或只移植子集导致与 PC 行为分叉。
- **唯一允许的变动**：为适配 EmuCoreX 架构而做的 **薄适配层微调**（例如：内存读写经 `MemoryInterface` 注入、头文件路径 / 命名空间、日志宏、条件编译、与 `VMManager` 生命周期对接方式等），不改变可观察的补丁语义。

### A.1 目标

- 与 PC 分支 **功能全集一致**：同一 pnach / 同一启用组合下，**加载结果、应用时机、持续/单次/组合 place、F 门与指针链** 等与 PC 对齐。
- **不要求**修改当前安卓「设定 → 金手指」Compose 界面；仅需保证 **核心 pnach 行为** 与 PC 一致。

### A.2 技术结论（避免走弯路）

- **不可**指望仅替换 `Patch.cpp` + `Patch.h` 两个文件完成移植：EmuCoreX 使用 **`MemoryInterface` + `ApplyPatch(ee, iop, …)`** 架构，与 PC 侧 **`ApplyLoadedPatches` / `ApplyPatch(p)`** 分叉；头文件中 **`PatchCommand` 布局**（含 `static_assert`）也可能与 PC 扩展字段不一致。
- **正确做法（全量）**：以 PC **`Patch.cpp` 全文为迁移蓝本**，在 EmuCoreX 工程中 **整文件逻辑对齐**（必要时拆分文件但须等价）；在 **内存访问、符号导出、include** 等处做 **架构适配**；保证 **`ApplyBootPatches` / `ApplyVsyncPatches` 与 PC 同序**，且 vsync 路径末尾含 **`ApplyFGateUpdates()`**、卸载路径含 **`ResetFGateState()`**（与 PC `Patch.cpp` 一致）。
- 验收标准：**同一组回归用例在 PC 与安卓上输出与游戏内效果一致**；差异仅允许来自浮点/未定义行为外的平台 bug，不允许「故意少实现」。

### A.3 建议任务拆分

| 序号 | 任务 | 说明 |
|------|------|------|
| A-1 | 全量对照合并 `Patch.cpp` | 按 PC 源码顺序核对，避免遗漏分支；指针链 / F 门 / extended / 平台补丁等 **全部** 覆盖 |
| A-2 | 对齐 `Patch.h` 与数据结构 | `PatchCommand` 等与 PC 一致；移除或更新阻碍迁移的断言 |
| A-3 | 适配 `MemoryInterface` | 将 PC 直接读写 EE/IOP 处改为经 `EEMemoryInterface` / `IOPMemoryInterface`，**不改写入语义** |
| A-4 | 生命周期与辅助模块 | `ReloadPatches`、热重载、`FindPatchFilesOnDisk`、与 `GameDatabase` / Achievements 等与 PC 对齐的调用关系 |
| A-5 | 全量回归测试 | 静态地址、多 place、指针链、值链、F 门、randint/float/swap 等 PC 已支持语法；与现有导入 pnach 流程 |

### A.4 参考源码（PC）

- `pcsx2/pcsx2/Patch.cpp`、`Patch.h`
- 调用链：`ApplyBootPatches` / `ApplyVsyncPatches` 内顺序与 `ApplyFGateUpdates`

## B. 贴图提取与替换：导航「快速操作」+ 专属 UI

### B.1 入口与导航

- 在 **主导航栏「快速操作」**（或现有 Quick Actions 等价入口）增加一项：**「贴图」**。
- 点击后 **弹出/导航至贴图专属界面**（全屏 Sheet、独立 Route 均可），**信息架构与视觉层级对标**现有「设定 → 金手指」页（卡片 + 主开关 + 说明 + 列表区），差异见下节文案与控件。

### B.2 UI 规格（对标金手指页，文案与控件映射）

以下将截图中的「金手指」语义替换为「贴图」，并 **新增两项**。

| 区域 | 对标原型的控件 | 贴图版要求 |
|------|----------------|------------|
| 顶栏 | 返回、标题、副标题、搜索/菜单（可选） | 标题区：**贴图**；副标题可为 **Texture replacements** 或与设置总览一致 |
| Tab 行（若保留） | 一般 / 影像 / 控制 / 档案 | 可与设置主 Tab 一致，或本页仅单页无 Tab（产品定） |
| 主标题 | 「金手指」 | **「贴图」** 或 **「贴图替换」** |
| 卡片 1 | 启用金手指 + 说明 | **启用贴图替换**（总开关）：对应 GS **加载替换纹理**（`LoadTextureReplacements` / 等价设置） |
| 卡片 2（新增） | — | **启用贴图提取**（总开关）：对应 **DumpReplaceableTextures**（及与 PC 一致的子项，如需要可折叠高级项） |
| 卡片 3（新增） | — | **提取目录**：展示当前路径 + **更改**；用户所选目录作为 **dump 输出根路径**（见 B.4） |
| 卡片 4 | 汇入金手指档案 | **导入贴图**：点击后 **弹窗要求输入游戏序列号**（Serial，如 `SLUS-xxx`），再打开系统文件夹选择器 |
| 说明文案 | pnach 命名提示 | 改为 **替换目录结构说明**（与 B.3 一致） |
| 列表区 | 已汇入的金手指档案 | **已导入的替换贴图**：列出当前游戏 **replacements** 目录下已识别的 **PNG / DDS**（仅文件名或含哈希摘要，与 PC 命名一致） |

**交互细节（导入）**

1. 用户输入 **游戏序列号**（需校验基本格式，与 `VMManager::GetDiscSerial()` 风格一致，大小写策略与 PC 目录一致）。
2. 用户选择 **文件夹** 后：将该文件夹内 **所有 `.png` / `.dds`（大小写不敏感）文件** **扁平复制** 到目标目录；**不要**复制用户选中的顶层文件夹本身作为一级子目录。
3. 复制完成后：**通知核心重载替换表**（见 B.5）。

### B.3 目录结构：与 PC / `GSTextureReplacements` 对齐

PC 侧逻辑（`GSTextureReplacements.cpp`）使用：

- 游戏目录：`Path::Combine(EmuFolders::Textures, s_current_serial)`
- 子目录：**`replacements`**（替换）、**`dumps`**（提取）

即物理路径为：

```text
{EmuFolders::Textures}/{Serial}/replacements/
{EmuFolders::Textures}/{Serial}/dumps/
```

**推荐安卓实现**：与上述 **完全一致**，这样 **无需改 `GetGameTextureDirectory()`** 即可复用 `GSTextureReplacements::ReloadReplacementMap()` 等逻辑。

> **关于需求文案「textures-游戏序列号-replacements」**  
> PCSX2 标准布局是 **`textures/<Serial>/replacements/`**（`textures` 为 `EmuFolders::Textures` 根），不是单层目录名 `textures-<Serial>-replacements`。若产品坚持字面命名，必须在 C++ 增加路径适配层（维护成本高）。**建议产品文案仍写「按游戏序列号分目录的 replacements」**，物理路径采用 PC 同款。

**列表数据来源**：扫描 `{EmuFolders::Textures}/{用户输入或当前光盘 Serial}/replacements/` 下 `*.png` / `*.dds`，排序与展示规则可与 PC 资源管理器一致。

### B.4 提取目录（用户配置）

- **含义**：启用「贴图提取」时，**可 dump 的贴图文件**写入位置的 **根** 仍由 `EmuFolders::Textures` + `Serial` + `dumps` 决定；若需 **完全自定义磁盘路径**（例如 SD 卡任意目录），有两种实现层级：
  - **轻量**：仅将 `Folders/Textures`（或安卓专用等价项）设为用户所选路径，使 **`dumps/` 与 `replacements/` 同根游戏子目录** 仍由核心创建（推荐，改动小）。
  - **重量**：为 `GSTextureReplacements::GetDumpFilename` 增加「dump 根覆盖」配置 — **仅在轻量方案不足时再做**。

开发团队应在技术方案中二选一并在本文档附录记录最终选项。

### B.5 原生层集成（禁止重复造轮子）

| 能力 | 复用点 |
|------|--------|
| 替换加载 / 哈希命名 | `GS/Renderers/HW/GSTextureReplacements.cpp`、配置项 `LoadTextureReplacements` |
| 提取 dump | `DumpReplaceableTextures` 及相关 GS 配置 |
| 重载列表 | `GSTextureReplacements::ReloadReplacementMap()`（VM 已存在调用链时可对齐触发时机） |
| 纹理根目录 | `EmuFolders::Textures`（`Pcsx2Config` / `Folders`） |

安卓侧通过现有 **`EmulatorBridge.setSetting`** 或 JNI 封装，将 UI 开关写入与 PC 相同的 **INI 键**；导入文件仅负责 **文件系统拷贝 + 通知 reload**。

### B.6 建议任务拆分

| 序号 | 任务 | 负责层 |
|------|------|--------|
| B-1 | 快速操作入口 + 导航 | Kotlin / Compose |

### B.7 验收清单（贴图）

- [ ] 打开「贴图」页，**启用替换** 后，已放入 `replacements` 的纹理在游戏中生效（与 PC 同 pnach/同 ISO 对比）。
- [ ] **启用提取** 后，dump 文件出现在 **`…/{Serial}/dumps/`**（或团队书面确认的等价路径）。
- [ ] **提取目录** 更改后，新 dump 落在预期根路径下。
- [ ] **导入贴图**：输入 Serial → 选文件夹 → 仅 **文件** 进入 `replacements`，无多余嵌套目录。
- [ ] 列表与磁盘内容一致；核心 **reload** 后无需重启即生效（若当前架构需重启，须在文档与 UI 说明）。
